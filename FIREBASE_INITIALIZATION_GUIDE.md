## 1. Required Firestore Collections (The Workspace Schema)

In Cloud Firestore, collections are created dynamically on the first document write. To ensure security rules, data subscriptions, and admin monitors resolve successfully, the following collections are used by the application:

1. **/school_settings**: Stores global institutional configuration (UDISE codes, Board IDs, Pass rates, Principal Welcome messages).
2. **/users**: Stores all validated user profiles (Students, Teachers, Admin, Visitors) indexed by role.
3. **/admins**: Stores secure, high-privilege configuration records used to authenticate admin functions.
4. **/notices**: Stores school circulars and exam schedules.
5. **/students**: Stores verified student academic records.
6. **/teachers**: Stores current verified faculty members.
7. **/attendance**: Stores transactional classroom attendance matrices.
8. **/examResults**: Stores academic performance tracking results.
9. **/homework**: Stores active daily homework assignments.
10. **/studyMaterials**: Stores digital notes and lesson plans.
11. **/books**: Stores library indices.
12. **/grievances**: Stores grievances submitted by students/parents.
13. **/certificates**: Stores certificates requested (Character Certificate, TC, etc.).

---

## 2. User Role Persistence

* **Primary Collection:** User roles are stored directly inside the **/users** collections under individual document profiles.
* **Secondary Ledger (Privilege Verification):** Administrator capabilities are additionally mirrored in the **/admins** collection to prevent privilege escalation or self-elevation attacks:
  - If a user claims `role: "ADMIN"` inside their `/users/{userId}` profile, security rules verify authorization by cross-checking the existence of a corresponding document under `/admins/{userId}`.

## 3. The First Admin Document Structure

To register the first Super Administrator, the system writes two individual documents mapping the authenticated user's unique identification key (`uid`):

### Document A: `/users/{uid}`
* **Collection:** `users`
* **Document ID:** Must be equal to the user's authentic `uid` generated by Firebase Authentication upon account creation.
* **Required Fields (TypeScript representation):**
  ```json
  {
    "id": "[AUTH_USER_UID]",
    "role": "ADMIN",
    "nameEn": "Smt. Aadiya Priyam (Admin)",
    "nameHi": "श्रीमती आदिया प्रियम् (प्रशासक)",
    "email": "omarbalika132@gmail.com"
  }
  ```

### Document B: `/admins/{uid}`
* **Collection:** `admins`
* **Document ID:** Must be equal to the user's authentic `uid` generated by Firebase Authentication.
* **Required Fields:**
  ```json
  {
    "uid": "[AUTH_USER_UID]",
    "role": "ADMIN",
    "permissions": ["all"]
  }
  ```

## 4. Bootstrap Programmatic Writes

When the programmatic **Initial School Setup Bootstrap** button is pressed, the client-side code (`src/AppContext.tsx`) triggers the following sequential writes:

* **[YES] Firebase Authentication User Creation:**
  - Attempts to create the credentials under `omarbalika132@gmail.com` with static default password `AdminPass123` via:
    `createUserWithEmailAndPassword(auth, 'omarbalika132@gmail.com', 'AdminPass123')`
  - *Fallback Handling:* If the authentication entry already exists, it signs in instead.
* **[YES] Firestore User Document Registration:**
  - Seeds the admin's profile doc under `/users/[UID]` matching the user's UID.
* **[YES] Firestore Admin Privilege Entry:**
  - Seeds the privilege authority document under `/admins/[UID]`.
* **[YES] School Settings Seeding (Implicit):**
  - While not written directly inside the `bootstrapFirstAdmin` click function, the active `onSnapshot` listener observing `/school_settings/current` automatically detects the absence of the config map on first launch and spawns the document with absolute defaults (`DEFAULT_SCHOOL_SETTINGS`) using:
    `setDoc(doc(firedb, 'school_settings', 'current'), DEFAULT_SCHOOL_SETTINGS)`

## 5. Clean Database Cold-Start Execution Sequence

When deploying the portal to a completely clean, empty project, the client-side subscriptions and security triggers orchestrate the following operational workflow:

```text
  [ START: Empty Firebase Instance ]
                 │
                 ▼
 1. Visitor opens page ──────────────────► App subscribes to '/school_settings/current'
                 │                         App subscribes to query('users', role == 'ADMIN')
                 ▼
 2. Monitor detects Empty Admin query ──► Sets 'isAdminCreated = false'
                 │
                 ▼
 3. Header modal renders Setup Alert ────► Displays: "SYSTEM BOOTSTRAP: No Admin Found"
                 │                         button: "🚀 Run Initial School Setup Bootstrap"
                 ▼
 4. Admin clicks Setup Bootstrap Button ─► Calls Auth 'createUserWithEmailAndPassword()'
                 │                         Creates '/users/[UID]' with "ADMIN" role
                 │                         Creates '/admins/[UID]' with "all" permissions
                 ▼
 5. Monitor detects Admin doc write ─────► Receives updated live snapshot and sets
                 │                         'isAdminCreated = true' (Bootstrap Alert disappears)
                 ▼
 6. Admin is now authenticated ──────────► Empty tables subscriptons trigger data seeding:
                                           automatically seeds notices, student rosters,
                                           calendars, mock performance reports.
                                           
  [ COMPLETE: School Portal Operational ]
```

## 6. Required Manual Operations (Before System Launch)

To transition from developmental code to a functional, secure high school production environment, standard administrators must complete the following 5 manual actions:

1. **Provision Firestore Database:**
   - Log in to the [Firebase Web Console](https://console.firebase.google.com/).
   - Click **Create Database**, select **Production Mode** (default secure read/write boundaries), and choose your geographic Google Cloud serving zone.
2. **Configure Authentication Email Provider:**
   - Go to **Authentication** -> **Sign-in Method** -> **Add New Provider**.
   - Enable **Email/Password** credentials (crucial for role-based system onboarding; students, parents, and teachers use these credentials generated during database CSV ingestion).
3. **Establish Environment Identity Files:**
   - Retrieve API coordinates from project settings inside the console.
   - Map keys into the root `/firebase-applet-config.json` inside the repository to ensure client-side calls establish connection to the proper Cloud environment.
4. **Deploy Security Rules:**
   - Copy-paste the highly secure `/firestore.rules` (enforcing Zero-Trust role validation check gates) into the **Rules** tab of Firestore inside the Firebase Console, or deploy automatically using the developer command tool.
5. **Execute First-Run Bootstrap:**
   - Load the portal domain URL, click **Portal Log In**, click the **Run Initial School Setup Bootstrap** banner to register the principal's official master user profile, and allow the system to populate initial timetables and performance metrics.
